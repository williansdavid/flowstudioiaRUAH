# FlowStudio AI — Arquitetura do Sistema
Data: 25 de junho de 2026
Stack: TanStack Start (SSR) + Router + React Query + TypeScript + Supabase + Tailwind + Vite + Netlify

---

## 1. Visão Geral
Plataforma white-label para studios de beleza (salões, barbearias, studios).
Cada studio tem deploy isolado: seu próprio Netlify, seu próprio Supabase, seu próprio banco, sua própria landing page e identidade visual.

### 1.1 Modelo de 2 Módulos
**Módulo 1 — Sistema** (núcleo compartilhado entre studios)
- `src/lib/`, `src/routes/` estruturais, `src/components/` core
- Auth SSR, supabase clients, query, utils genéricos, guards, types do banco

**Módulo 2 — Site** (individual por studio)
- `src/sites/<slug>/`
  - `config/` (branding, identity, content, businessHours, seo)
  - `components/` (landing)
  - `utils/` (fachadas)
  - `types/`
  - `studio.ts` (entry point)

**Regra de ouro:**
- `src/lib/` nunca importa de `src/sites/`
- `src/sites/` consome `src/lib/core/` via fachadas (`buildBrandingCss.ts`, `buildSeo.ts`)

---

## 2. Stack (travada)
- **Framework:** TanStack Start (SSR)
- **Roteamento:** TanStack Router
- **Estado servidor:** TanStack React Query
- **Linguagem:** TypeScript (`noUncheckedIndexedAccess: true`)
- **Banco:** Supabase (Postgres + Auth + RLS)
- **Estilização:** Tailwind CSS
- **Build/Bundler:** Vite
- **Deploy:** Netlify (1 por studio)
- **Ícones:** lucide-react
- **Notificações:** sonner
- **Validação:** zod
- **Animações:** framer-motion
- **UI Componentes:** radix-ui (Dialog), Button próprio, ConfirmDialog, WhatsAppButton

**Proibido:** microsserviços, multi-tenant, dados compartilhados, Kubernetes, filas, CQRS, event sourcing.

## 4. Features (módulos)

### 4.1 admin-shell
Layout administrativo compartilhado com navegação por seções e controle de roles.
- `AdminLayout.tsx` — Layout principal com sidebar + topbar
- `Sidebar.tsx` — Navegação lateral
- `SidebarItem.tsx` — Item de navegação
- `Topbar.tsx` — Barra superior
- `PlaceholderScreen.tsx` — Tela placeholder para rotas em construção

- `ClientCombobox.tsx` — Combobox de seleção de cliente
- `QuickClientModal.tsx` — Modal de cadastro rápido de cliente
- `SlotsStep.tsx` — (legado) Step de seleção de horário

**BookingWizard** (novo fluxo de criação):

- `BookingWizard/BookingWizardFooter.tsx` — Navegação (voltar/continuar/confirmar) com sticky bottom

**Server functions:**

### 4.7 sales (PDV)
Módulo de Ponto de Venda. Dois fluxos de entrada: via agendamento (staff clica "Concluir" e vai para o PDV com o serviço bloqueado) ou autônomo (carrinho vazio, adiciona produtos avulsos).

**Layout:** Página separada `/admin/pdv`, duas colunas (catálogo à esquerda, carrinho à direita). Inspirada em maquininha de cartão.

**Componentes:**
- `PdvPage.tsx` — Página principal do PDV. Gerencia estado do carrinho, split de pagamento, fluxo de finalização. Recebe `appointmentId` opcional via search params.
- `ProductGrid.tsx` — Grid de produtos (catálogo) com busca/filtro
- `ProductCard.tsx` — Card de produto com clique + animação de voo pro carrinho (framer-motion layoutId)
- `CartPanel.tsx` — Carrinho (coluna direita). Mostra itens, quantidades, total acumulado
- `CartItem.tsx` — Item do carrinho com quantidade, valor unitário, total por item. Botão de remover desabilitado para itens `is_locked=true` (staff não remove serviço do agendamento)
- `SplitPayment.tsx` — Seção de split de pagamento. Botão "Adicionar pagamento" → dropdown com métodos ativos + input de valor. Barra de progresso "Faltam R$ XX,00". Valida soma dos splits == total.
- `PaymentRow.tsx` — Linha de método + valor no split
- `PdvHeader.tsx` — Header com info do agendamento (cliente, serviço, valor) quando veio de agendamento

**Server functions:**
- `createSale.ts` — Finaliza venda: insere `sale_payments`, atualiza `sales` para `completed`, gera `finance_transactions` (type='income', category='product', amount=total, reference_id=sales.id), atualiza appointment para `completed` se veio de agendamento. Valida soma dos splits no servidor.
- `createSaleDraft.ts` — Cria rascunho da venda (`sales` com status='draft' + `sale_items`). Se existir rascunho pendente do mesmo usuário, retoma em vez de criar novo.
- `listProducts.ts` — Lista produtos ativos
- `listPaymentMethods.ts` — Lista formas de pagamento ativas (ordenadas por sort_order)
- `getSaleDraft.ts` — Busca rascunho pendente do usuário logado (por created_by + status='draft')
- `getAppointmentSaleData.ts` — Dados do agendamento para o PDV (cliente, serviço, valor)

**Hooks:**
- `useCreateSale` — Mutation para finalizar venda
- `useCreateSaleDraft` — Mutation para criar/retomar rascunho
- `useProducts` — Query para listar produtos ativos
- `usePaymentMethods` — Query para listar formas de pagamento
- `useSaleDraft` — Query para buscar rascunho pendente
- `useAppointmentSaleData` — Query para dados do agendamento (enabled apenas quando appointmentId presente)

**Types (`types.ts`):**
- `PaymentMethod` — `id, name, icon, isActive, sortOrder`
- `ProductItem` — `id, name, price, avatarUrl, isActive`
- `SaleItem` — `id, saleId, itemType, itemId, itemName, quantity, unitPrice, totalPrice, isLocked`
- `Sale` — `id, saleType, appointmentId, clientName, totalAmount, status, createdBy`
- `SplitPayment` — `paymentMethodId, amount`
- `SaleDraft` — `saleId, items, totalAmount`

**Regras de negócio:**
- Staff não remove item locked (serviço do agendamento). Admin pode tudo.
- Soma dos splits deve ser exatamente igual ao total. Validado client + server.
- Ao finalizar, gerar `finance_transactions` com `type='income'`, `category='product'`, `amount=total_amount`, `reference_id=sales.id`
- Ao finalizar PDV vindo de agendamento, atualizar appointment para `completed` com `payment_method_id` do primeiro split
- Venda finalizada não pode ser editada — apenas cancelada (status='cancelled')
- Rascunho (status='draft') criado na abertura do PDV. Se existir rascunho pendente, carregar ao invés de criar novo

## 11. Convenções e Padrões

### 11.1 Feature-based Architecture
Cada feature em `src/features/<feature>/` é autocontida:
  /
  components/     Componentes React
  hooks.ts        Hooks + mutations/queries
  server/         Server functions (createServerFn)
  types.ts        Tipos da feature
  utils/          Utilitários
  index.ts        Barrel público (opcional)

### 11.2 Fluxo de trabalho (engenharia)
1. **Investigar** — validar código real / API
2. **Confirmar** — resumir entendimento
3. **Propor** — abordagem + impacto + riscos
4. **Aguardar OK** — só executar após autorização
5. **Entregar** — código completo, tipado, loading/error/empty states, SSR-safe, mobile-first

### 11.3 Regras de UI
- **Mobile-first** — toda interface prioriza mobile
- Componentes com loading, error, empty states
- Modal com proteção contra perda de dados não salvos (beforeunload + confirm)
- Animações com framer-motion (spring, staggerChildren)
- Botões com `active:scale-95` para feedback tátil

### 11.4 Tipagem estrita
`noUncheckedIndexedAccess: true` no tsconfig — todo acesso a array exige verificação de undefined.

### 11.5 Query Keys (convenção)
- `['appointments', 'day', date]` — Agendamentos do dia
