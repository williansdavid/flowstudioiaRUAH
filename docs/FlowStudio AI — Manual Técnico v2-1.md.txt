1. VISÃO GERAL E MODELO DE NEGÓCIO
================================================================================

O FlowStudio AI é uma plataforma white-label de alto desempenho, projetada
especificamente para atender às necessidades de studios de beleza, barbearias
e centros de estética. O sistema opera sob um modelo de ISOLAMENTO TOTAL POR
CLIENTE, garantindo máxima segurança, performance e personalização para cada
studio [1].

Cada studio cliente do FlowStudio AI possui uma instância completamente
dedicada e isolada, composta por:

• Deploy Independente: Cada studio tem seu próprio deploy no Netlify,
  assegurando autonomia e performance otimizada.
• Banco de Dados Dedicado: Um projeto isolado no Supabase, incluindo
  PostgreSQL, autenticação (Auth) e Row Level Security (RLS).
• Identidade Visual Personalizada: Configurações de branding e landing pages
  exclusivas em src/sites/<slug>/, permitindo identidade própria [1][2].

1.1 Estrutura de Módulos

O sistema é arquitetado em dois núcleos fundamentais:

1. Sistema (Core): Localizado em src/lib/ e src/features/, contém a lógica
   de negócio compartilhada, componentes de interface administrativa e
   integrações de infraestrutura. Representa o núcleo comum.
2. Site (Studio): Localizado em src/sites/<slug>/, gerencia a presença
   digital pública de cada studio e consome o core através de fachadas.

REGRADE OURO DA MODULARIDADE:
É estritamente proibido que src/lib/ importe diretamente de src/sites/.
Em vez disso, src/sites/ DEVE consumir funcionalidades de src/lib/core/
exclusivamente via fachadas como buildBrandingCss.ts e buildSeo.ts [2].

2. ARQUITETURA GERAL
================================================================================

Fluxo de dados: Internet → Netlify SSR → TanStack Start + Router →
React Query + Server Functions → Supabase (PostgreSQL, Auth, Storage, RLS)

Componentes:
• Netlify SSR: Hospedagem com suporte a Server-Side Rendering para SEO e
  performance. Cada studio tem deploy independente.
• TanStack Start + Router: Roteamento tipado e gerenciamento de estado.
• React Query + Server Functions: Camada de abstração para API e mutações.
• Supabase: Backend-as-a-Service (PostgreSQL, Auth, Storage, RLS).

3. STACK TECNOLÓGICA
================================================================================

3.1 Tecnologias Core

• Framework: TanStack Start (SSR) + TanStack Router + TanStack React Query
• Backend: Supabase (PostgreSQL, Auth, Storage, Row Level Security)
• Linguagem: TypeScript com noUncheckedIndexedAccess: true
• Estilização: Tailwind CSS + Framer Motion + Radix UI
• Build/Ferramentas: Vite + PostCSS
• Infraestrutura: Netlify (SSR Functions + deploy automatizado)
• Utilitários: lucide-react (ícones), sonner (toasts), zod (validação)

3.2 Adições v2.0 e v2.1 (mantidas)

• Zustand: Gerenciamento de estado global leve para UI
• Date-fns: Manipulação e formatação de datas
• Recharts: Visualização de dados e gráficos
• IDB (IndexedDB): Cache local para persistência offline

3.3 Tecnologias Proibidas

• Redux (substituído por React Query/Zustand)
• Axios (substituído por fetch nativo em Server Functions)
• Bibliotecas de UI pesadas sem suporte a Tailwind
• Microsserviços, multi-tenant de banco único, Kubernetes, filas de
  mensagens complexas, CQRS, Event Sourcing

12. CONVENÇÕES E PADRÕES
================================================================================

12.1 Arquitetura Baseada em Features

Cada feature em src/features/<nome>/ segue:
  components/     Componentes React da feature
  hooks.ts        Hooks personalizados (React Query)
  server/         Server Functions específicas
  types.ts        Tipos TypeScript
  utils/          Utilitários
  index.ts        Barrel export

12.2 Regras de Engenharia

• Sem acesso direto ao Supabase via React (sempre createServerFn)
• SQL inline proibido (toda query via Server Functions)
• Features autocontidas (cada uma gerencia seus próprios dados)
• Invalidar Query Keys após mutations
• Mobile First (toda interface responsiva)
• SSR Safe (código executa em servidor e cliente)
• Tipagem 100% (TypeScript strict, sem any)
• Zod para validação de schemas em Server Functions

12.3 Convenção de Query Keys

Padrão hierárquico para invalidação precisa:
  ['appointments', 'list', { date, staffId }]
  ['appointments', 'detail', { appointmentId }]
  ['staff', 'list']
  ['staff', 'workingHours', { staffId }]
  ['sales', 'draft', { appointmentId }]
  ['services', 'list']
  ['dashboard', 'data', { period }]

12.4 Regras de UI

• Mobile-first: toda interface responsiva
• Estados obrigatórios: loading (skeleton/spinner), erro (mensagem + retry),
  vazio (empty state)
• beforeunload: alerta ao fechar modal com dados não salvos
• framer-motion: animações suaves
• active:scale-95: feedback tátil em botões

13. FLUXOS DE TRABALHO
================================================================================

13.1 Fluxo de Agendamento

1. Cliente acessa BookingWizard (ou admin agenda manualmente)
2. Passo 1: Seleciona serviço (listActiveServices)
3. Passo 2: Seleciona profissional (listBookableStaff)
4. Passo 3: Escolhe data/horário (getAvailableSlots — calcula slots com
   base em working hours e time off)
5. Passo 4: Confirma dados (createAppointment com validação de conflito)
6. Agendamento aparece na Agenda do admin (getDayAppointments)
7. Profissional realiza atendimento
8. Admin conclui agendamento e abre PDV
9. PDV: adiciona produtos, aplica split de pagamento
10. createSale finaliza venda → gera finance_transactions
11. Appointment status alterado para completed
12. Trigger handle_appointment_lifecycle dispara eventos CRM

13.2 Fluxo do PDV

1. Agendamento concluído ou venda avulsa
2. Carrinho: adicionar produtos/serviços
3. Aplicar descontos se necessário
4. SplitPayment: dividir valor entre formas de pagamento
5. Confirmar venda (createSale)
6. createSaleDraft para rascunhos
7. Transação financeira registrada em finance_transactions
8. Estoque de produtos atualizado
9. Status do appointment atualizado para completed
10. CRM Log gerado para registro

13.3 Fluxo do CRM

1. Evento detectado (appointment no_show, aniversário, cliente sem retorno
   há 45+ dias)
2. Evento registrado em client_events
3. Motor de tarefas do CRM avalia o evento
4. Tarefa gerada e registrada em crm_logs
5. Notificação enviada ao admin
6. Admin executa ação (contato, oferta, etc.)
7. Tarefa marcada como resolvida
8. Dashboard atualizado com métricas de CRM
