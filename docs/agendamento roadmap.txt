5. FLUXO DO CADASTRO PELO PORTAL (A1)
----------------------------------------------------------------
Cliente preenche: nome, telefone, email (obrigatorio),
                  aniversario (opcional)
  |
  +-- telefone OU email JA existe como ACCOUNT (tem login)
  |     -> "Voce ja tem conta" -> recuperar senha (decisao #4)
  |
  +-- telefone JA existe como WALKIN (sem login)
  |     -> "Esse telefone ja esta cadastrado. Fale com a gente"
  |        -> botao wa.me com mensagem pre-preenchida
  |
  +-- nada existe -> cria conta nova -> confirmacao Supabase
                     -> cliente define senha

Mensagem pre-preenchida do wa.me:
"Ola! Quero agendar um horario e nao consegui pelo portal. Pode
buscar um horario pra mim e liberar meu cadastro para agendamento
pelo portal? Meu nome e [nome], telefone [telefone]."

----------------------------------------------------------------
6. PROCEDIMENTOS DO ADMIN (X2)
----------------------------------------------------------------
X2.1 — CADASTRAR WALK-IN
  - Admin abre /clients -> "Novo cliente".
  - Preenche: full_name, phone (obrigatorio), email (opcional),
    aniversario (opcional).
  - Sistema grava em `clients` SEM profile_id (sem conta, sem auth).
  - Telefone normalizado canonico (+55DDDNUMERO) na gravacao (item 1.3).
  - Resultado: ficha apta a agendamentos presenciais, sem login.

X2.2 — CONVERTER WALK-IN EM CONTA
  Gatilho: cliente pede acesso ao portal (via WhatsApp do passo 5)
  OU admin decide liberar.

  Pre-condicao: clients.email PREENCHIDO e valido. Se nao tiver,
                admin coleta e atualiza a ficha antes.

  Passos:
  1. Admin abre a ficha em /clients.
  2. Botao "Convidar para o portal" (visivel so se email preenchido
     e profile_id ainda NULL).
  3. Sistema chama Supabase inviteUserByEmail(email) com metadata
     contendo o client_id alvo.
  4. Supabase cria auth.user (sem senha) e envia email com link de
     definicao de senha.
  5. Cliente clica, define a propria senha, confirma email.
  6. Trigger handle_new_user detecta metadata.client_id e LIGA:
     - cria profiles (role=client)
     - UPDATE clients SET profile_id = novo profile
       (NAO cria clients novo — usa o existente, preserva historico).
  7. Resultado: walk-in vira account, historico intacto.

  Admin NUNCA conhece/define a senha. Seguranca preservada.
